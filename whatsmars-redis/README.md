# [Redis](https://github.com/antirez/redis)

## Redis集群方案
目前国内 Redis 集群的主流方案主要分为三大类，其中 **Redis Cluster（官方原生集群）** 是目前生产环境中最主流的首选方案。

以下是目前国内主流的 Redis 集群方案及其特点：

### 1. 官方原生 Redis Cluster（生产主流首选）
这是 Redis 3.0 之后官方推出的原生分布式集群方案，也是目前互联网和大型企业项目中最广泛采用的标准方案。
* **核心特点**：采用无中心化的 P2P 架构，通过 16384 个哈希槽（SLOT）进行数据动态分片，节点间通过 Gossip 协议通信。
* **优势**：无代理层，性能损耗极低；原生支持自动故障转移和在线扩缩容（Resharding）；官方持续维护，支持最新特性。
* **局限**：需要客户端支持集群协议（Smart Client），有一定侵入性；不支持跨槽的多 Key 操作。
* **适用场景**：高并发、大数据量、需要水平扩展的生产环境。

### 2. 云托管集群方案（企业级推荐）
随着云原生架构的普及，国内各大云厂商（如阿里云、腾讯云等）提供的托管版 Redis Cluster 已成为企业生产环境的重要选择。
* **核心特点**：底层基于原生 Redis Cluster 或深度定制的分布式架构（如阿里云 Tair、腾讯 Tendis），对外提供标准化的集群服务。
* **优势**：提供自动故障转移、自动备份、一键在线扩容等能力，具备 SLA 保证；大幅降低企业的运维成本，部分平台还引入了 AI 智能体进行根因分析与风险自动处置。
* **局限**：成本相对较高，底层架构不可控，某些高级功能可能受限。
* **适用场景**：希望免去底层运维烦恼、追求高可用与稳定性的中大型企业。

### 3. 主从复制 + Sentinel（哨兵）（中小型系统标配）
虽然严格意义上它不是“分布式分片集群”，但它是国内中小型系统解决 Redis 高可用最成熟、最常见的方案。
* **核心特点**：1个主节点 + N个从节点，配合奇数个 Sentinel 节点进行监控与自动故障转移。
* **优势**：部署简单，成熟稳定，对客户端几乎零侵入。
* **局限**：写操作仍是单点，无法突破单机内存上限，不支持自动水平扩容。
* **适用场景**：QPS 在 2~3 万以内、数据量在几十 GB 以内、读多写少且不需要横向扩展的业务。

### 4. 代理型集群（Codis / Twemproxy）（历史方案/特定场景）
在 Redis Cluster 成熟之前，这类方案曾在国内非常流行，但目前整体处于维护放缓或逐渐被替代的状态。
* **Twemproxy**：轻量级代理，静态分片，不支持动态扩缩容，适合中小规模静态集群。
* **Codis**：企业级动态分片代理，支持可视化 Dashboard 和 1024 个 SLOT 动态迁移，客户端零侵入。但由于其基于旧版 Redis 二次开发，版本滞后且组件复杂，目前新项目已较少采用。

**总结建议**：
如果是新建的互联网项目或中大型系统，直接首选 **Redis Cluster** 或 **云托管 Redis 集群**；如果是中小型业务或读多写少场景，**主从 + 哨兵** 依然是性价比最高的选择。

## 缓存三大经典问题
这是面试和实战中的高频考点，必须掌握解决方案：
- 缓存穿透：查询不存在的数据，绕过缓存直击 DB。解决：布隆过滤器拦截 + 缓存空值。
- 缓存击穿：热点 Key 过期瞬间大量请求涌入。解决：互斥锁（Redisson 分布式锁）+ 逻辑过期。
- 缓存雪崩：大量 Key 同时过期。解决：过期时间加随机值 + 多级缓存架构 + 缓存预热。

## Thread or epoll
MySQL为什么使用线程的模型而不是epoll之类的技术？
MySQL的主要任务是把数据组织成树表，在磁盘和内存之间进行转换，并且进行大量的查找和排序，所有这些操作花费的时间都很长。
假设每次操作平均占用30ms，而通过网络返回只需要1ms，则总花费31ms。时间主要花费在CPU和内存占用，而不是网络等待上。
所以，非阻塞io没有优势。而应用服务器，主要是接入客户端，对客户端认证，然后通过远程访问数据库，之后就是等待期，
应用服务器大部分时间花费在等待数据库结果以及各种远程服务比如某种密钥计算或者认证。那么Redis为什么用非阻塞？
主要是Redis的数据目标是存储到内存，而且没有大密度查找排序这种计算，所以Redis的计算很快，也就不存在io问题。
